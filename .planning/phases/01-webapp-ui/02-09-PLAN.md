<objective>
Add backend data model and API support for the node graph visualizer. This plan lays the foundation that the frontend graph component (02-10) will consume.

Three gaps in the current data model prevent graph visualization:
1. **No character-to-node mapping** — StoryAnalysis.cast extracts character names but they're ephemeral (in-memory only, lost on refresh) and not linked to individual nodes.
2. **No analysis persistence** — StoryAnalysis lives only in generationState.lastAnalysis on the frontend. Switching stories or refreshing loses it.
3. **No branch switching API** — The DB schema supports branching (parent_id + position), but there's no endpoint to switch active_path to follow a different branch, and no tree-structured endpoint for the graph to consume.

This plan adds: character_mentions table, node_analyses table, character extraction in the generation pipeline, a tree endpoint, and a branch-switching endpoint.

Inspired by: narrative graph models (paragraphs-as-nodes, character supernodes) from maetl.net and Ian Pears' Arcadia (branching narrative map with character strands).
</objective>

<task id="1" title="Create migration 002: character_mentions and node_analyses tables">
<action>
1. Create `backend/app/db/migrations/002_graph_support.sql`
2. Add `character_mentions` table:
   - `id TEXT PRIMARY KEY` (UUID)
   - `node_id TEXT NOT NULL REFERENCES nodes(id) ON DELETE CASCADE`
   - `character_name TEXT NOT NULL` — normalized name extracted from cast
   - `role TEXT` — optional role descriptor (e.g., "protagonist", "antagonist")
   - `UNIQUE(node_id, character_name)` — one mention per character per node
3. Add `node_analyses` table:
   - `id TEXT PRIMARY KEY` (UUID)
   - `node_id TEXT NOT NULL REFERENCES nodes(id) ON DELETE CASCADE`
   - `analysis_json TEXT NOT NULL` — full StoryAnalysis serialized as JSON
   - `created_at TEXT NOT NULL DEFAULT (datetime('now'))`
   - `UNIQUE(node_id)` — one analysis per node (latest wins)
4. Add indexes: `idx_char_mentions_node ON character_mentions(node_id)`, `idx_char_mentions_name ON character_mentions(character_name)`, `idx_node_analyses_node ON node_analyses(node_id)`
5. Verify the migration runner in database.py auto-discovers and runs numbered migration files in order
</action>

<task id="4" title="Persist analysis and character mentions in WebSocket generation flow">
<action>
1. In `ws.py`, after `run_cycle()` completes and the draft content is saved, persist the analysis:
   - Insert into `node_analyses` (node_id = draft_id, analysis_json = result.analysis.model_dump_json())
   - Call `extract_characters(result.analysis.cast)`
   - Insert each (character_name, role) into `character_mentions` for the draft node
2. This happens in the same DB transaction as the draft content update (after streaming completes)
3. On `accept`: character_mentions and node_analyses persist (they reference the node, which persists)
4. On `reject`: character_mentions and node_analyses cascade-delete with the node
</action>

<task id="6" title="Add branch switching endpoint">
<action>
1. Add `PATCH /api/stories/{story_id}/active-path` endpoint accepting `SwitchPathRequest`
2. Given a `target_node_id`:
   - Verify the node exists and belongs to the story
   - Walk up the tree from target_node_id via parent_id to build the path from root to target
   - Reverse the path to get root → ... → target order
   - Update `stories.active_path` to this new path
   - Update `stories.updated_at`
3. Return the updated `StoryResponse` with new active_path and all nodes
4. This enables the frontend to click any node in the graph and switch the editor to show that branch
</action>
