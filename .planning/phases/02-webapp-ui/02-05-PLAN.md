<objective>
Build the WebSocket-based AI generation pipeline with inline draft flow in the editor.

Purpose: This is the core AI writing experience — the user clicks "Generate" and AI-generated text streams token-by-token directly into the Tiptap editor, visually distinguished with white text (ai_generated provenance mark). The user can cancel mid-stream, accept the draft (persisting it as a real node), or reject it (removing it from the editor). The WebSocket connection handles bidirectional communication: server streams tokens, client sends control messages (generate, cancel, accept, reject).

Output: A working end-to-end generation flow: click Generate → see tokens stream into editor with ai_generated color → accept/reject the draft. Draft nodes created in SQLite during generation with is_draft=1.
</objective>

  <action>
    Build the WebSocket endpoint on the backend and a reconnecting WebSocket client on the frontend.

    **Backend — `02-worktrees/webapp-ui/backend/app/routers/ws.py`:**

    Create a WebSocket endpoint at `/ws/generate` using FastAPI's WebSocket support. The endpoint handles a message loop with these message types:

    1. **`generate` message** (client → server):
       - Payload: `{ type: "generate", story_id: string, node_id: string }` — node_id is the parent node to continue from
       - Server flow:
         a. Load the story and walk the active_path to build the full story text (concatenate all node contents along the path)
         b. Load the current LLM config from the module-level state (set via REST in Plan 04's llm router)
         c. Create a draft node in SQLite: `INSERT INTO nodes (id, story_id, parent_id, position, content, node_type, source, is_draft) VALUES (uuid, story_id, node_id, next_position, '', 'paragraph', 'ai_generated', 1)`
         d. Send `{ type: "draft_created", node_id: draft_node_id }` to client
         e. Call `run_cycle()` from story service — this calls `parse_structured()` for StoryAnalysis, then `parse_structured()` for StoryContinue
         f. Note: `run_cycle()` is NOT streaming — it returns the full result at once. To simulate streaming, after `run_cycle()` completes, stream the `draft_next` text character-by-character with small delays (10ms per char). This gives the visual effect of streaming while using the structured output API.
         g. For each chunk, send `{ type: "token", content: chunk }` and update the draft node's content in SQLite
         h. After all tokens sent, send `{ type: "complete", node_id: draft_node_id, analysis: analysis_dict }` with the full StoryAnalysis
         i. Store the analysis alongside the story (or in the response — Plan 06 will display it)

    2. **`cancel` message** (client → server):
       - Set a cancellation flag that the token-streaming loop checks
       - Send remaining content so far as a partial draft
       - Send `{ type: "cancelled", node_id: draft_node_id }`

    3. **`accept` message** (client → server):
       - Payload: `{ type: "accept", node_id: string, content: string, provenance_spans: [...] }`
       - Update the draft node: `is_draft = 0`, update content with final editor content
       - Update story's `active_path` to include the accepted node
       - Save provenance spans from the editor
       - Send `{ type: "accepted", node_id: string }`

    4. **`reject` message** (client → server):
       - Payload: `{ type: "reject", node_id: string }`
       - Delete the draft node from SQLite
       - Send `{ type: "rejected", node_id: string }`

    Handle `WebSocketDisconnect` gracefully. Wrap all DB operations with `get_db()` context manager.

    Update `02-worktrees/webapp-ui/backend/app/main.py` to include the WebSocket router: `app.include_router(ws_router)`.

    **Frontend — `02-worktrees/webapp-ui/frontend/src/lib/api/ws.ts`:**

    Create a WebSocket client class with reconnection:

    ```ts
    type WSMessageType = 'generate' | 'cancel' | 'accept' | 'reject';
    type WSServerMessage = { type: 'token', content: string }
      | { type: 'draft_created', node_id: string }
      | { type: 'complete', node_id: string, analysis: StoryAnalysis }
      | { type: 'cancelled', node_id: string }
      | { type: 'accepted', node_id: string }
      | { type: 'rejected', node_id: string }
      | { type: 'error', message: string };

    class WebSocketClient {
      private ws: WebSocket | null = null;
      private reconnectAttempts = 0;
      private maxReconnectAttempts = 5;
      private onMessage: ((msg: WSServerMessage) => void) | null = null;
      connectionState: 'disconnected' | 'connecting' | 'connected' = 'disconnected';

      connect(onMessage: (msg: WSServerMessage) => void): void { ... }
      send(msg: object): void { ... }
      disconnect(): void { ... }
      // Auto-reconnect with exponential backoff (1s, 2s, 4s, 8s, 16s)
    }
    export const wsClient = new WebSocketClient();
    ```

    Use the Vite proxy — connect to `ws://${window.location.host}/ws/generate` (NOT localhost:8000 directly). The Vite proxy in vite.config.ts already forwards `/ws` to the backend.

    ```ts
      error = $state<string | null>(null);

      hasDraft = $derived(this.status === 'draft_ready');

      startGeneration(storyId: string, nodeId: string) { ... }
      appendToken(content: string) { ... }
      completeGeneration(nodeId: string, analysis: StoryAnalysis) { ... }
      cancelGeneration() { ... }
      acceptDraft() { ... }
      rejectDraft() { ... }
    }
    export const generationState = new GenerationState();
    ```

    Wire up `generationState` methods to `wsClient` message handling — when the WebSocket receives a `token` message, call `generationState.appendToken()`, etc.
